## 一、Java并发编程

### Q1: 在HIS项目中用多线程优化护士拆分速度，具体说说场景、线程池配置和并发问题？

在HIS 3.0产品中，护士执行医嘱拆分时，一条长期医嘱可能需要拆分成几十条甚至上百条临时的执行记录（比如每天3次、连开7天的药品），涉及大量数据库写入。原来的方式是单线程逐条插入，在高频场景下（如上午集中处理时段）护士操作明显卡顿。

**线程池配置：**

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    4,                    // corePoolSize
    8,                    // maximumPoolSize  
    30L,                  // keepAliveTime
    TimeUnit.SECONDS,
    new LinkedBlockingDeque<>(100),  // 有界队列
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
);
```

线程数设置依据：医嘱拆分主要是数据库IO操作，属于 **IO密集型**，参考公式 `核心线程数 = CPU核数 / (1 - 阻塞系数)`。服务器4核，阻塞系数按0.8估算约20线程。但要结合HikariCP连接池最大连接数（默认10），最终core设4、max设8。

**并发问题的处理：**
- 用 `CountDownLatch` 等待所有拆分任务完成后再提交事务
- 每条拆分记录带一个 `batchId`，部分失败也能回滚整批
- 用 `ThreadLocal` 传递 traceId，确保日志链路可追踪

### Q2: synchronized vs ReentrantLock + AQS原理 + 分布式锁方案

**synchronized 和 ReentrantLock 核心区别：**

| 对比维度 | synchronized | ReentrantLock |
|---------|-------------|---------------|
| 实现层面 | JVM关键字，monitorenter/monitorexit | JDK API |
| 可中断 | 不支持 | 支持 lockInterruptibly() |
| 超时获取 | 不支持 | 支持 tryLock(timeout) |
| 公平性 | 非公平 | 支持公平/非公平 |
| 条件等待 | wait/notify | Condition.await/signal |
| JDK6后性能 | 引入偏向锁/轻量级锁，差距缩小 | — |

**AQS原理（AbstractQueuedSynchronizer）：**

AQS内部维护一个 `volatile int state` + CLH双向队列。

ReentrantLock.lock() 流程：
1. CAS尝试将 state 从0改为1，成功则获得锁
2. 失败则入队到CLH队列尾部，通过 `LockSupport.park()` 挂起
3. 前驱节点释放锁后 unpark() 唤醒后继

```java
// 非公平锁核心
final boolean nonfairTryAcquire(int acquires) {
    final Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {
        if (compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(current);
            return true;
        }
    } else if (current == getExclusiveOwnerThread()) {
        // 可重入
        int nextc = c + acquires;
        setState(nextc);
        return true;
    }
    return false;
}
```

**分布式锁方案对比：**

| 方案 | 优点 | 缺点 |
|-----|------|------|
| Redis SETNX | 简单、性能高 | 主从切换可能丢锁 |
| Redisson RLock | 看门狗自动续期 | — |
| Redlock | 多节点容错 | 时钟漂移争议 |
| ZooKeeper | CP强一致 | 性能略低 |
| etcd | 强一致 + watch | 生态相对新 |

**幂等性方案核心代码：**

```java
String requestId = UUID.randomUUID().toString();
Boolean locked = redisTemplate.opsForValue()
    .setIfAbsent(key, requestId, 30, TimeUnit.SECONDS);
if (locked) {
    try {
        // 业务处理
    } finally {
        // Lua脚本保证原子性
        String lua = "if redis.call('get',KEYS[1]) == ARGV[1] then redis.call('del',KEYS[1]) end";
        redisTemplate.execute(new DefaultRedisScript<>(lua, Long.class), 
            Arrays.asList(key), requestId);
    }
}
```

### Q3: 线上CPU飙高/线程阻塞排查方法

排查步骤：

1. `top -H -p <pid>` 找CPU最高的线程ID
2. 转16进制：`printf '%x\n' <tid>`
3. `jstack <pid> | grep -A 30 <十六进制tid>` 定位堆栈
4. 频繁GC用 `jstat -gcutil <pid> 1000 10` 观察GC频率
5. `jmap -dump:live,format=b,file=heap.hprof <pid>` 抓堆快照，用MAT分析

**实际案例**：HIS项目现场卡顿排查发现大量线程阻塞在数据库连接获取上——慢SQL占满连接池，加复合索引后解决。

### Q4: 线程池参数设置方法

**核心公式：**

- CPU密集型：`核心线程数 = CPU核数 + 1`
- IO密集型：`核心线程数 = CPU核数 × 2` 或 `CPU核数 / (1 - 阻塞系数)`
  - 阻塞系数一般在0.8~0.9

**队列选择：**

| 队列 | 场景 | 注意事项 |
|-----|------|---------|
| LinkedBlockingQueue | 通用 | 生产环境必须用有界队列 |
| SynchronousQueue | 内部处理线程多 | newCachedThreadPool 使用 |
| ArrayBlockingQueue | 有界、固定容量 | — |

**拒绝策略：**

| 策略 | 行为 | 场景 |
|-----|------|------|
| AbortPolicy | 抛异常 | 不推荐 |
| CallerRunsPolicy | 提交线程自己执行 | 自然限流，推荐 |
| DiscardPolicy | 静默丢弃 | 可丢失的任务 |
| DiscardOldest | 丢弃最老任务 | — |

## 二、Spring/MyBatis框架原理

### Q5: Spring IoC容器 refresh() 流程

核心流程：

```
prepareRefresh()
  → obtainFreshBeanFactory()       // 解析配置，生成BeanDefinition
  → invokeBeanFactoryPostProcessors() // 调用BeanFactoryPostProcessor
  → registerBeanPostProcessors()     // 注册BeanPostProcessor
  → initMessageSource()
  → initApplicationEventMulticaster()
  → registerListeners()
  → finishBeanFactoryInitialization() // 实例化所有非懒加载单例Bean
  → finishRefresh()
```

**三级缓存解决循环依赖：**

```java
// DefaultSingletonBeanRegistry
Map<String, Object> singletonObjects           // 一级缓存：成品
Map<String, Object> earlySingletonObjects       // 二级缓存：半成品
Map<String, ObjectFactory<?>> singletonFactories // 三级缓存：工厂
```

A依赖B、B依赖A的场景：
1. A创建时提前暴露 `ObjectFactory` 到三级缓存
2. B创建发现依赖A，从三级缓存拿到A的提前引用（调用 `getEarlyBeanReference()` 处理AOP）
3. B完成创建，A完成属性注入

为什么三级而不是二级：AOP需要——三级缓存存的是 `ObjectFactory`，在真正暴露时才判断是否需要生成代理对象，避免所有Bean提前走AOP。

### Q6: MyBatis四大对象 + 拦截器实现多租户

**四大核心对象：**

| 对象 | 职责 |
|-----|------|
| Executor | SQL执行 + 一级缓存管理 |
| StatementHandler | Statement创建 + 参数设置 |
| ParameterHandler | 参数绑定 |
| ResultSetHandler | 结果映射 |

**拦截器原理：** MyBatis通过JDK动态代理对四大对象创建代理，拦截点如 Executor.update/query、StatementHandler.prepare 等。

**多租户拦截器实现：**

```java
@Intercepts({
    @Signature(type = StatementHandler.class, method = "prepare", 
               args = {Connection.class, Integer.class})
})
public class TenantInterceptor implements Interceptor {
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        StatementHandler handler = (StatementHandler) invocation.getTarget();
        String sql = handler.getBoundSql().getSql();
        String tenantId = TenantContextHolder.getTenantId();
        
        if (sql.contains("FROM") && !sql.contains("tenant_id")) {
            sql = sql.replace("WHERE", 
                "WHERE tenant_id = '" + tenantId + "' AND ");
        }
        // 通过MetaObject反射修改SQL
        MetaObject metaObject = SystemMetaObject.forObject(handler);
        metaObject.setValue("delegate.boundSql.sql", sql);
        return invocation.proceed();
    }
}
```

### Q7: @Transactional 失效场景

1. **同类中方法自调用**（不走代理）
   ```java
   // 错误
   public void methodA() { this.methodB(); }
   @Transactional public void methodB() {}
   
   // 正确：注入自己
   @Autowired private UserService self;
   public void methodA() { self.methodB(); }
   ```

2. 非public方法上无效
3. 数据库表不支持事务（MySQL MyISAM）
4. 异常被catch没抛出去
5. 异常类型不对：默认只回滚 `RuntimeException` 和 `Error`，需指定 `rollbackFor = Exception.class`

## 三、Redis深度

### Q8: Redis线程模型 — 6.0多线程IO

**Redis高性能原因：**
- 纯内存操作
- 非阻塞IO（epoll）
- 单线程避免上下文切换和锁竞争

**Redis 6.0变化：** 引入多线程IO处理网络读写：
- 6.0前：读socket → 解码 → **执行命令** → 编码 → 写socket，全在主线程
- 6.0后：读socket/解码、编码/写socket交给IO线程池，**命令执行仍是单线程**
- 效果：网络包大、客户端多时显著提升吞吐，命令执行顺序性和原子性不受影响

### Q9: Redis分布式锁 — 主从切换丢锁问题

**问题场景：** 客户端A在master获取锁，master未同步到slave就宕机，slave升为master，客户端B也能获取到同一个锁。

**解决方案：**

| 方案 | 说明 |
|-----|------|
| Redlock | 向N个独立节点依次申请，超半数成功才算获取。但有时钟漂移争议 |
| Redisson看门狗 | 自动续期，避免锁提前释放 |
| 数据库兜底 | 配合唯一约束或乐观锁做双重校验 |
| 本地消息表 | 最终一致性保障 |

### Q10: 多级缓存架构 + Redis宕机降级

**多级缓存架构：**

```
浏览器 → CDN → Nginx/Lua缓存 → 本地缓存(Caffeine) → Redis集群 → MySQL
```

**缓存更新策略：** Canal监听MySQL binlog → MQ广播 → 各服务本地缓存刷新

**Redis宕机降级方案：**
1. 本地缓存(Caffeine)兜底
2. Sentinel限流 + 降级配置
3. 数据库连接池自然限流
4. 跨机房Redis主从 + 哨兵自动切换

## 四、消息中间件（重点）

### Q11: 消息中台架构设计

**架构分层：**

```
接入层 → 消息路由层 → 渠道适配层 → 第三方通道
```

六大渠道：站内信、短信、微信模板消息、APP推送、AI语音、企业微信

**统一消息模型：**

```json
{
  "msgId": "唯一ID",
  "templateCode": "模板编码",
  "channels": ["SMS", "WECHAT"],
  "params": {"name": "张三"}
}
```

**核心设计：**
- 路由规则：按消息类型配置，支持动态调整和灰度
- 渠道适配器模式：每个渠道实现统一 `MessageSender` 接口
- 消息去重：唯一msgId，消费端幂等
- 失败策略：重试3次 → 死信队列 → 人工补偿或定时重推

### Q12: RabbitMQ保证消息不丢失（全链路）

**生产者 → Broker：**
- 开启 confirm 机制：等待Broker ack回调，nack或超时重发
- `mandatory=true` + ReturnCallback：消息不可路由时回调

**Broker持久化：**
- `durable=true` 声明队列持久化
- `deliveryMode=2` 消息持久化
- 镜像队列（或Quorum队列）多副本

**Broker → 消费者：**
- 关闭autoAck，使用手动ack
- 业务成功后ack，失败则nack(requeue=true/false)

**代码示例：**

```java
// 生产端
rabbitTemplate.setConfirmCallback((correlationData, ack, cause) -> {
    if (!ack) {
        // 记录到数据库，定时任务补偿
    }
});

// 消费端
@RabbitListener(queues = "order.queue")
public void handle(Message msg, Channel channel) {
    try {
        process(msg);
        channel.basicAck(msg.getMessageProperties().getDeliveryTag(), false);
    } catch (Exception e) {
        channel.basicNack(tag, false, requeue);
    }
}
```

### Q13: RabbitMQ延时队列 vs RocketMQ延时消息

**RabbitMQ：** 利用死信队列（DLQ）机制
1. 消息设置TTL（`x-message-ttl`）
2. 队列设置死信转发（`x-dead-letter-exchange`）
3. TTL到期后自动转发到死信队列，消费者监听

**缺点：** 同一队列不同TTL消息会被前面的堵塞，不支持精确秒级延时。

**RocketMQ：**
- 内置18个延时等级（1s/5s/10s/30s/1m/2m...2h）
- 设置 `setDelayTimeLevel(level)`
- Broker写入 `SCHEDULE_TOPIC_XXXX`，后台定时恢复写入原Topic
- **优势：** 毫秒级精度、高吞吐

**选型建议：** 高精度大规模延时消息用RocketMQ，简单延时场景RabbitMQ够用。

### Q14: 消息四个核心问题解决方案

| 问题 | 解决方案 |
|------|---------|
| 重复消费 | 消费侧幂等（唯一ID去重表）、数据库唯一索引 |
| 消息丢失 | confirm机制 + 持久化 + 手动ack + 本地消息表补偿 |
| 顺序性 | 同一业务ID路由到同一Queue、单消费者 |
| 大量积压 | 增加Queue/Cosumer并发、新建临时队列批量处理、排查消费者瓶颈 |

### Q16: 服务治理（Sentinel）

**限流：**
- QPS限流：核心接口配置QPS阈值
- 线程数限流：慢接口防护
- 热点参数限流：针对特定资源ID

**熔断：** 错误率/慢调用比例超阈值 → 熔断 → 降级

**降级：** 非核心功能高峰期降级、缓存不可用时限流+降级

**核心理念：** 限流是预防，熔断是止损，降级是保核心。

## 六、JVM调优

### Q18: HIS系统线上JVM调优案例

**现象：** 上午高峰期系统卡顿，医生操作慢。

**排查过程：**
1. `top -H` 发现CPU不高，大量D状态线程
2. `jstack` 找到阻塞在数据库连接上的线程
3. `jstat -gcutil` 显示FGC每几分钟一次
4. `jmap` + MAT分析：HashMap无限制缓存撑爆老年代

**优化：**
```bash
-Xms4g -Xmx4g -Xmn1.5g -XX:+UseG1GC 
-XX:MaxGCPauseMillis=200 -XX:InitiatingHeapOccupancyPercent=45
-XX:G1HeapRegionSize=4m
```
- Caffeine替代HashMap，设置最大条目和过期时间

**结果：** FGC从3分钟/次降到几乎没有，响应从3-5秒降到1秒内。

### Q19: G1垃圾收集器原理

**核心概念：**
- **Region**：堆分多个大小相等的Region（1-32MB），可扮演Eden/Survivor/Old/Humongous
- **RSet（RememberSet）**：记录Region被谁引用，避免全堆扫描
- **SATB**：并发标记时保证快照一致性

**GC流程：**
1. Young GC：STW，复制存活对象到Survivor
2. 并发标记：和业务线程并发
3. Mixed GC：老年代Region + 年轻代一起回收
4. Full GC：退化SerialOld

**G1 vs CMS vs ZGC：**

| GC | 特点 | 适用 |
|----|------|------|
| CMS | 并发标记清除 | 低延迟，但碎片化 |
| G1 | 可预测停顿 | 大堆 > 4GB |
| ZGC | 染色指针，几乎无停顿 | JDK17+，超大堆 |

## 七、内存泄漏排查与解决

### Q20: 内存泄漏问题如何排查与解决？

#### 内存泄漏 vs 内存溢出

| 概念 | 区别 |
|-----|------|
| 内存泄漏（Memory Leak） | 不再使用的对象仍然被GC Roots引用，无法被回收，堆占用持续增长 |
| 内存溢出（OOM） | 堆内存已满且GC无法腾出空间，直接抛出 `OutOfMemoryError` |

内存泄漏是导致OOM的主要原因之一。

#### 常见内存泄漏场景

| 类型 | 典型表现 |
|-----|---------|
| 集合类无限制增长 | `HashMap`/`ArrayList` 作为静态字段或缓存不断put但不清除 |
| 未关闭的资源 | 数据库连接、IO流、`OkHttpClient` 等未显式 close |
| 内部类/匿名类持有外部引用 | 非静态内部类隐式持有外部类引用，导致外部类无法回收 |
| ThreadLocal 未 remove | 线程池中线程复用，ThreadLocal 中的对象一直残留在线程的 ThreadLocalMap 中 |
| 注册的监听器/回调未移除 | addListener 后没有 removeListener |
| 类加载器泄漏 | Tomcat 热部署时自定义类加载器无法释放 |

#### 排查方法论（7步法）

```
1. 发现现象    →  应用响应慢/频繁FGC/OOM
2. 确认症状    →  jstat -gcutil 看FGC频率和堆占用趋势
3. 抓取快照    →  jmap -dump:live,format=b,file=heap.hprof <pid>
4. 分析对象    →  MAT 找 Dominator Tree → GC 根路径 → 确认为什么未被回收
5. 定位代码    →  根据引用链定位到具体的业务代码位置
6. 修复/验证   →  补上释放逻辑或改用弱引用，上线后对比
7. 建立防护    →  CI流水线集成内存泄漏检测、配置GC预警
```

#### 实战案例一：ThreadLocal 内存泄漏（多租户上下文）

**场景：** 微信小程序中登录用户的租户上下文通过 ThreadLocal 传递，高并发下老年代持续增长。

**排查过程：**

1. 通过 `jstat -gcutil` 发现老年代占用持续上升，每次FGC后回收很少
2. `jmap dump` 后用MAT打开，Dominator Tree 中大量 `TenantContext` 对象
3. 查看 GC Root 路径：`Thread → ThreadLocalMap → Entry → TenantContext`
4. 确认漏洞：使用了 `static ThreadLocal<TenantContext>`，但在请求结束后没有调用 `remove()`

**根因：** 线程池中的线程被复用，每个线程的 `ThreadLocalMap` 中 entry 的 key 虽然是 WeakReference，但 value 是强引用链。线程不销毁，value 永远不会被回收。

**修复方案：**

```java
// 错误：只清除了值，但entry还在
try {
    // 业务处理
} finally {
    TenantContextHolder.clear(); // 仅做了 context = null
}

// 正确：必须 remove
try {
    // 业务处理
} finally {
    TenantContextHolder.remove(); // threadLocal.remove()
}
```

配合 Filter 统一处理：

```java
public class TenantCleanupFilter implements Filter {
    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) {
        try {
            chain.doFilter(request, response);
        } finally {
            TenantContextHolder.remove();
            MDC.clear();
        }
    }
}
```

#### 实战案例二：静态集合类缓存泄漏（HIS项目）

**场景：** HIS系统的药品字典数据使用 `static HashMap` 缓存，上午高峰期医生频繁开医嘱，系统变得越来越慢，最终 OOM。

**排查过程：**

1. 监控告警触发 OOM，重启后恢复，但几天后又OOM
2. 添加 `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/heap.hprof`
3. OOM后自动 dump，用MAT打开分析
4. Dominator Tree 中 `HashMap` 占用了80%的堆内存
5. 查看 GC Root：`Class → static field → HashMap`
6. HashMap 中存在大量 `DrugInfo` 对象和 `String` key
7. 代码确认是每次查询新的医嘱项时都往这个 HashMap 中put，从未清理

**修复方案：**

```java
// 原代码
private static Map<String, DrugInfo> drugCache = new HashMap<>();
public DrugInfo getDrug(String code) {
    if (!drugCache.containsKey(code)) {
        drugCache.put(code, dao.queryByCode(code));
    }
    return drugCache.get(code);
}

// 修复后：用Caffeine替代，限制容量和过期时间
private static Cache<String, DrugInfo> drugCache = Caffeine.newBuilder()
    .maximumSize(5000)
    .expireAfterWrite(1, TimeUnit.HOURS)
    .recordStats()
    .build();

public DrugInfo getDrug(String code) {
    return drugCache.get(code, dao::queryByCode);
}
```

#### 实战案例三：消息中台 WebSocket 连接泄漏

**场景：** 消息中台使用 WebSocket 实现实时推送，线上运行几天后服务端响应变慢，连接数异常升高。

**排查过程：**

1. `netstat -anp | wc -l` 发现连接数远超预期
2. 查看 `Endpoint` 管理类，发现 `ConcurrentHashMap` 中存储的 Session 数量异常
3. 打印 Session 状态：大量已关闭但未从 map 中移除的 Session

**根因：** WebSocket `onClose` 回调中移除 Session 的逻辑被 catch 住了未正常执行，导致 Session 虽然已断开但仍留在管理 map 中。消息推送时遍历所有 Session 发送消息，对已关闭的 Session 发送失败后也没有清理。

**修复方案：**

```java
@OnClose
public void onClose(Session session) {
    // 确保清理逻辑一定执行
    try {
        String userId = (String) session.getUserProperties().get("userId");
        SessionManager.removeSession(userId, session.getId());
    } catch (Exception e) {
        log.error("移除WebSocket session异常", e);
        // 兜底：按sessionId直接移除
        SessionManager.removeBySessionId(session.getId());
    }
}

// 发送消息时增加清理逻辑
public void sendMessage(String userId, String message) {
    List<Session> sessions = SessionManager.getSessions(userId);
    Iterator<Session> it = sessions.iterator();
    while (it.hasNext()) {
        Session session = it.next();
        try {
            session.getBasicRemote().sendText(message);
        } catch (IOException e) {
            log.warn("Session已失效，移除: {}", session.getId());
            it.remove(); // 遍历中清理失效session
        }
    }
}
```

### Q2: Spring AI 的 Structured Output（结构化输出）你怎么理解的？

**你就这么说：**

"就是说，以前调大模型返回的是 JSON 字符串，还得自己解析。Spring AI 的 Structured Output 能做到**直接帮你转成 Java 对象**。

概念很简单——在调用时指定个 BeanOutputConverter：

```java
@GetMapping("/ai/doctor")
public DoctorInfo doctor(@RequestParam String name) {
    BeanOutputConverter<DoctorInfo> converter = 
        new BeanOutputConverter<>(DoctorInfo.class);
    
    String output = chatClient.prompt()
        .user("请介绍一下" + name + "医生")
        .call()
        .content();
    
    return converter.convert(output);
}
```

### Q3: RAG（检索增强生成）的原理你理解吗？你怎么搭的 Demo？

用户问题 → (1)检索知识库 → (2)拼接到 Prompt → (3)大模型回答

@Bean

### Q2: 抖音open_api的access_token为什么要缓存到Redis？过期了怎么处理？

**基于代码的回答：**

"看代码里的`DouyinLocalLifeAuthorizationServiceImpl.getAccessToken()`：

```
redisKey = DOUYIN_LOCAL_LIFE_ACCESS_TOKEN_PREFIX + clientKey
先从Redis查 → 有就返回 → 没有再调用抖音API获取 → 存Redis(过期时间-60s缓冲)
```

**为什么Redis缓存：**
1. access_token 2小时有效期，每次调API都去刷新会频繁请求抖音接口，可能触发限流
2. 多个线程并发获取token时，用Redis的原子性保证只调一次抖音
3. 缓存key按商户维度隔离，不同商家不同token

**过期处理：**
- 获取时先Redis，过期了自动调抖音重新获取
- 调用API时如果返回token_expired，立即清除Redis缓存，下次请求自动重新获取
- 参考抖音文档说的："重复获取access-token后会使上次的失效，但有5分钟缓冲"

**这个经验在巨鲨出海可以直接复用：**
Google Ads API、Facebook Marketing API 一样有access_token有效期，一样的缓存策略——无非client_id/client_secret `换成` app_id/app_secret。"

### Q11: 集森项目里用到OAuth2.0对接抖音，说说整个授权流程？你在实现中踩过什么坑？

**基于代码的回答：**

"**授权流程（以抖音web-app为例）：**

```
1. 用户点击"授权抖音" → 302到抖音授权页（scope=video.publish,data.external）
2. 用户确认 → 302回调到我们的redirect_uri，带上code
3. DouyinWebAppAuthorizationServiceImpl 拿code换access_token
4. Redis缓存access_token（2h有效期，提前5分钟刷新）
5. 用access_token调用视频发布、数据查询API
```

**踩过的坑：**

1. **access_token并发刷新**：多个请求同时发现token过期，同时去刷新，导致一个token失效。解决：用Redis分布式锁，只让一个线程去刷新

2. **scope权限不够**：上线后发现拿不到用户数据，因为授权时scope没传全。解决：授权URL增加 `scope=video.list,video.data,user.info`

3. **抖音沙箱数据问题**：沙箱环境返回的数据字段和线上不一致。解决：线上先灰度验证

4. **抖音API限流**：每秒调用次数有限，并发高了报 `exceeded rate limit`。解决：令牌桶限流 + 失败重试"

   "我想了解一下，公司目前在医疗信息化领域主要聚焦在哪些业务方向？未来的发展规划是什么？"

   "目前技术团队的规模和分工是怎样的？我这个岗位进来后主要负责哪些业务模块？"

3. **关于个人成长：**
   "公司对技术人员有什么培养规划或者晋升通道吗？"

4. **关于入职（加分项）：**
   "如果我能有幸入职，在入职前有没有什么资料或者系统我可以提前了解一下？"

### Q10: 如果你接到offer，多久能入职？

"如果确定录用，我可以在收到offer后**1-2周内**到岗入职。因为我目前已经离职或者在交接阶段（根据实际情况说），时间上比较灵活。"

> 1. **着装**：虽然是线上面试，建议穿衬衫或Polo衫，不要穿T恤
> 2. **环境**：找一个安静、光线好的地方，背景不要太乱
> 3. **语速**：比平时稍慢一点，声音洪亮，显得自信
> 4. **直视摄像头**：让老板感觉你在和他对视
> 5. **准备好纸笔**：如果老板问技术问题，可以简单画图辅助表达
> 6. **松弛感**：三面老板一般不会故意刁难你，放松表现自己就好
